# Migrating from LangChain.js

This guide maps LangChain.js concepts to Reactive Agents equivalents. If you have existing agents built with LangChain.js, this will help you translate your mental model.

## Concept Mapping

| LangChain.js | Reactive Agents | Notes |
|---|---|---|
| `AgentExecutor` | `ReactiveAgent` (via builder) | Builder pattern instead of constructor |
| `Tool` / `StructuredTool` | `ToolDefinition` | Declarative schema + handler function |
| `LLMChain` / `SequentialChain` | Reasoning strategy | ReAct, Plan-Execute, ToT, Reflexion, Adaptive |
| `BufferMemory` / `ConversationBufferMemory` | Working memory (tier: "standard") | In-memory conversation history |
| `VectorStoreRetrieverMemory` | Semantic memory (tier: "enhanced") | SQLite + embeddings |
| `ChatPromptTemplate` | System prompt + persona | `.withPersona()` or `.withSystemPrompt()` |
| `CallbackManager` | Lifecycle hooks | 10-phase hooks via `.withHook()` |
| `OutputParser` | `AgentResult.output` | Structured via `final-answer` tool |
| `ChatOpenAI` / `ChatAnthropic` | `.withProvider()` + `.withModel()` | 6 providers, 40+ models via LiteLLM |

Reactive Agents provides 4 memory layers automatically:
- **Working memory** — current conversation context (like BufferMemory)
- **Semantic memory** — vector-indexed knowledge (like VectorStoreRetrieverMemory)
- **Episodic memory** — past execution experiences
- **Procedural memory** — learned tool-use patterns

## Key Differences

1. **Explicit reasoning control** — LangChain uses implicit agent loops; Reactive Agents exposes 5 named strategies with per-strategy configuration
2. **10-phase lifecycle** — every execution has observable, hookable phases (bootstrap through complete)
3. **Effect-TS composition** — services compose via layers, not dependency injection or constructor chaining
4. **Built-in safety** — guardrails, kill switch, behavioral contracts, and budget enforcement are first-class
5. **Local-first** — Ollama provider works out of the box with model-adaptive context profiles

import type { ToolDefinition, ToolParameter } from "./types.js";

/**
 * Fluent builder for creating ToolDefinition objects.
 *
 * Provides a chainable API that is more ergonomic than constructing
 * raw ToolDefinition objects, while producing the same output format.
 *
 * @example
 * ```typescript
 * const tool = new ToolBuilder("analyze")
 *   .description("Analyze a dataset")
 *   .param("url", "string", "Dataset URL", { required: true })
 *   .param("format", "string", "Output format", { enum: ["json", "csv"] })
 *   .handler(async (params) => ({ result: "..." }))
 *   .build();
 * ```
 */
export class ToolBuilder {
  private _name: string;
  private _description?: string;
  private _parameters: ToolParameter[] = [];
  private _returnType?: string;
  private _category?: "search" | "file" | "code" | "http" | "data" | "custom";
  private _riskLevel: "low" | "medium" | "high" | "critical" = "low";
  private _timeoutMs: number = 30_000;
  private _requiresApproval: boolean = false;
  private _source: "builtin" | "mcp" | "function" | "plugin" = "function";
  private _isCacheable?: boolean;
  private _cacheTtlMs?: number;
  private _handler?: (params: Record<string, unknown>) => unknown | Promise<unknown>;

  constructor(name: string) {
    this._name = name;
  }

  /** Set the tool description shown to the LLM. */
  description(desc: string): this {
    this._description = desc;
    return this;
  }

  /**
   * Add a parameter to the tool.
   *
   * @param name - Parameter name
   * @param type - JSON Schema type
   * @param description - Human-readable description for the LLM
   * @param options - Optional: required (default: false), default value, enum choices
   */
  param(
    name: string,
    type: "string" | "number" | "boolean" | "object" | "array",
    description: string,
    options?: {
      required?: boolean;
      default?: unknown;
      enum?: string[];
    },
  ): this {
    const param: ToolParameter = {
      name,
      type,
      description,
      required: options?.required ?? false,
      ...(options?.default !== undefined ? { default: options.default } : {}),
      ...(options?.enum ? { enum: options.enum } : {}),
    };
    this._parameters.push(param);
    return this;
  }

  /** Set the human-readable return type description. */
  returns(returnType: string): this {
    this._returnType = returnType;
    return this;
  }

  /** Set the tool category for discovery and filtering. */
  category(cat: "search" | "file" | "code" | "http" | "data" | "custom"): this {
    this._category = cat;
    return this;
  }

  /** Set the risk level (default: "low"). */
  riskLevel(level: "low" | "medium" | "high" | "critical"): this {
    this._riskLevel = level;
    return this;
  }

  /** Set the execution timeout in milliseconds (default: 30000). */
  timeout(ms: number): this {
    this._timeoutMs = ms;
    return this;
  }

  /** Set whether human approval is required before execution (default: false). */
  requiresApproval(required: boolean): this {
    this._requiresApproval = required;
    return this;
  }

  /** Set the tool source (default: "function"). */
  source(src: "builtin" | "mcp" | "function" | "plugin"): this {
    this._source = src;
    return this;
  }

  /** Set whether results can be cached (default: undefined, cache service decides). */
  cacheable(cacheable: boolean): this {
    this._isCacheable = cacheable;
    return this;
  }

  /** Set custom cache TTL in milliseconds. */
  cacheTtl(ms: number): this {
    this._cacheTtlMs = ms;
    return this;
  }

  /** Attach a handler function for tool execution. */
  handler(fn: (params: Record<string, unknown>) => unknown | Promise<unknown>): this {
    this._handler = fn;
    return this;
  }

  /** Retrieve the attached handler (useful for registration). */
  getHandler(): ((params: Record<string, unknown>) => unknown | Promise<unknown>) | undefined {
    return this._handler;
  }

  /**
   * Build the ToolDefinition.
   *
   * @throws Error if name is empty or description is missing
   */
  build(): ToolDefinition {
    if (!this._name || this._name.trim() === "") {
      throw new Error("ToolBuilder: name is required and cannot be empty");
    }
    if (!this._description) {
      throw new Error("ToolBuilder: description is required — call .description() before .build()");
    }

    return {
      name: this._name,
      description: this._description,
      parameters: this._parameters,
      riskLevel: this._riskLevel,
      timeoutMs: this._timeoutMs,
      requiresApproval: this._requiresApproval,
      source: this._source,
      ...(this._returnType !== undefined ? { returnType: this._returnType } : {}),
      ...(this._category !== undefined ? { category: this._category } : {}),
      ...(this._isCacheable !== undefined ? { isCacheable: this._isCacheable } : {}),
      ...(this._cacheTtlMs !== undefined ? { cacheTtlMs: this._cacheTtlMs } : {}),
    };
  }
}
```

# Production Deployment Checklist

Reactive Agents is designed for production use. This checklist covers the safety, reliability, cost, and observability features you should enable before deploying.

## Security

- [ ] **Enable guardrails** — `.withGuardrails()` activates injection detection, PII detection, and toxicity filtering on all inputs
- [ ] **Set tool risk levels** — mark destructive tools as `riskLevel: "high"` or `"critical"` and enable `requiresApproval: true`
- [ ] **Use behavioral contracts** — `.withBehavioralContracts()` enforces rules like "never execute code without user approval"
- [ ] **Enable identity** — `.withIdentity()` adds Ed25519 agent certificates and RBAC for multi-agent setups
- [ ] **Restrict tool access** — use `.withTools({ allowedTools: [...] })` to whitelist only the tools each agent needs
- [ ] **Sandbox code execution** — use Docker sandbox for `code-execute` tool in production: `.withTools({ dockerSandbox: true })`

**What happens if you skip it:** Without guardrails, prompt injection attacks can make your agent execute unintended tools. Without tool restrictions, agents have access to all registered tools including file system operations.

## Reliability

- [ ] **Set max iterations** — `.withMaxIterations(N)` prevents runaway loops (default: 10)
- [ ] **Enable kill switch** — always available via `agent.stop()` and `agent.terminate()`, but ensure your application has a way to trigger these
- [ ] **Configure timeouts** — `.withTimeout(ms)` sets a hard execution time limit
- [ ] **Set retry policy** — `.withRetryPolicy({ maxRetries: 2, backoffMs: 1000 })` handles transient LLM failures
- [ ] **Monitor loop detection** — the kernel runner automatically detects stuck loops (repeated tools, repeated thoughts, consecutive thoughts without action) and fails gracefully

**What happens if you skip it:** Without iteration limits, a confused agent can loop indefinitely, consuming tokens. Without timeouts, a hung LLM connection blocks your application thread.

## Cost Control

- [ ] **Enable cost tracking** — `.withCostTracking()` tracks token usage and estimated cost per execution
- [ ] **Set budget limits** — `.withCostTracking({ budget: { perRequest: 0.50, daily: 10.00 } })` hard-stops execution when limits are hit
- [ ] **Use appropriate model tiers** — not every task needs GPT-4 or Claude Opus; use cost routing to match task complexity to model capability
- [ ] **Monitor token usage** — `result.metadata.tokensUsed` and the metrics dashboard show per-execution consumption

**What happens if you skip it:** A single poorly-scoped task can consume thousands of tokens (and dollars) if the agent loops through expensive multi-step reasoning.

## Observability

- [ ] **Enable observability** — `.withObservability({ verbosity: "normal" })` shows the metrics dashboard on every execution
- [ ] **Use hooks for monitoring** — `.withHook("complete", "after", handler)` for execution completion notifications
- [ ] **Enable live streaming** — `.withObservability({ live: true })` streams phase events during execution for real-time monitoring
- [ ] **Track debriefs** — agent results include `result.debrief` with structured summaries, key findings, and lessons learned

**What happens if you skip it:** Without observability, you have no visibility into why an agent took 30 seconds or used 5,000 tokens. Debugging production issues becomes guesswork.

## Error Handling

- [ ] **Set global error handler** — `.withErrorHandler((error, context) => { ... })` catches all execution errors with phase context
- [ ] **Handle FiberFailure** — the builder's `agent.run()` already unwraps Effect-TS errors into clean Error objects with suggestions
- [ ] **Check error suggestions** — use the `errorContext(error)` standalone function to extract remediation hints (Data.TaggedError objects are frozen and can't have methods added)
- [ ] **Enable graceful shutdown** — call `agent.terminate()` on process exit to cleanly stop any running executions

**What happens if you skip it:** Errors surface as nested stack traces with no actionable information. Users must read source code to diagnose failures.

# Working with Sub-Agents

Reactive Agents supports two types of sub-agents: **static** (declared at build time) and **dynamic** (spawned at runtime by the LLM). This guide covers what context is forwarded, how to control sub-agent behavior, and known limitations.

## Static vs Dynamic Sub-Agents

| Feature | Static (`.withAgentTool()`) | Dynamic (`spawn-agent` tool) |
|---|---|---|
| Declared at | Build time | Runtime (by LLM) |
| Tool whitelist | `tools: [...]` in config | `tools` parameter in spawn call |
| Persona | `persona: { role, tone, ... }` | LLM provides `role`, `instructions`, `tone` |
| Max iterations | Hardcoded or from config | Default from framework |
| Use when | You know the sub-tasks ahead of time | Agent discovers delegation needs dynamically |

### Decision Tree

```
Is the sub-task known at build time?
  YES -> Use static sub-agent (.withAgentTool())
    Is the sub-agent task-specific?
      YES -> Whitelist specific tools
      NO  -> Let it inherit parent tools
  NO  -> Use dynamic sub-agent (spawn-agent tool)
    Is the task simple (1-2 tools)?
      YES -> Provide tools list in spawn params
      NO  -> Let framework auto-scope
```

## What Context IS Forwarded

When a sub-agent is spawned, the parent forwards a **context prefix** as the sub-agent's system prompt. This includes:

1. **Parent tool results** — the most recent tool results from the parent's execution, summarized
2. **Parent working memory** — relevant entries from the parent's conversation history
3. **Task description** — the specific instruction given to the sub-agent

**Maximum context size:** 2,000 characters (configurable via `MAX_PARENT_CONTEXT_CHARS` constant in `agent-tool-adapter.ts`)

### Example of forwarded context

```
[Parent Context]
Recent tool results:
- web-search("TypeScript trends"): Found 5 articles about TypeScript 5.x adoption...
- file-read("./data/metrics.json"): { "users": 1523, "growth": "12%" }

Working memory:
- User asked for a quarterly report on TypeScript ecosystem

[Sub-Agent Task]
Analyze the metrics data and write a summary paragraph.
```

## What Is NOT Forwarded

- **Scratchpad entries** — parent scratchpad data is not shared with sub-agents
- **Full tool result history** — only recent results within the character budget
- **Episodic/semantic memory** — sub-agents start with fresh memory
- **In-flight reasoning state** — sub-agents don't see the parent's thought chain

## Workarounds for Context Limitations

### Strategy 1: Write key findings to the task instruction

Instead of relying on automatic context forwarding, include critical information directly in the sub-agent task:

```typescript
const agent = await ReactiveAgents.create()
  .withProvider("anthropic")
  .withReasoning()
  .withAgentTool({
    name: "data-analyst",
    description: "Analyze structured data and produce summaries",
    tools: ["file-read", "code-execute"],
    persona: { role: "data analyst", tone: "precise" },
  })
  .build();

// The parent agent naturally includes context when delegating:
// "Analyze the data in ./metrics.json. Key context: user count is 1523,
//  growth is 12%, and we need a quarterly summary format."
```

### Strategy 2: Use shared files as context bridge

Have the parent write findings to a file that the sub-agent reads:

```typescript
// Parent writes context
// -> file-write("./scratch/context.md", "Key findings: ...")
// Sub-agent reads it
// -> file-read("./scratch/context.md")
```

## Tool Whitelisting

Control which tools a sub-agent can access:

### Static sub-agents
```typescript
.withAgentTool({
  name: "researcher",
  description: "Research a topic using web search",
  tools: ["web-search", "http-get"],  // Only these tools available
})
```

### Dynamic sub-agents
The parent LLM can specify tools when spawning:
```
spawn-agent({"task": "Search for TypeScript trends", "tools": ["web-search"]})
```

### Always-included tools
Some tools are always available to sub-agents regardless of whitelist:
- `scratchpad-write` / `scratchpad-read` — for intermediate storage
- `context-status` — for checking context usage
- `final-answer` — for completing the task

These are defined in `ALWAYS_INCLUDE_TOOLS` in the agent-tool-adapter.

## Sub-Agent Personas

Personas let you steer sub-agent behavior:

```typescript

  description: "Review code for security vulnerabilities",
  persona: {
    role: "Security auditor",
    background: "Expert in OWASP Top 10, CWE, and secure coding practices",
    instructions: "Focus on injection, auth, and data exposure risks. Rate severity as Critical/High/Medium/Low.",
    tone: "direct and technical",
  },

```

## Known Limitations

1. **Context truncation** — parent context is truncated at 2,000 characters. For data-heavy tasks, use file-based context sharing.
2. **No scratchpad forwarding** — sub-agents cannot read the parent's scratchpad entries.
3. **Small model performance** — models under 14B parameters struggle with sub-agent delegation. The sub-agent may hallucinate tool results or fail to follow the task.
4. **Overhead for simple tasks** — delegation adds ~4x overhead (extra LLM calls for spawning + sub-agent reasoning). Only delegate when the sub-task genuinely benefits from isolation.
5. **No cross-agent memory** — each sub-agent starts fresh. Cross-agent learning requires the experience store (`.withExperienceLearning()`).

description: Decision tree and performance characteristics for the 5 built-in reasoning strategies

# Choosing a Reasoning Strategy

Reactive Agents ships with 5 reasoning strategies. Each is optimized for a different task profile. This guide helps you pick the right one.

## Quick Decision Tree

```
Does the task need tools?
  NO  -> No reasoning needed (direct LLM call via agent.chat())
  YES ->
    Is it a single-step tool call?
      YES -> ReAct (simple, fast)
      NO  ->
        Does the task have a clear multi-step plan?
          YES -> Plan-Execute-Reflect
          NO  ->
            Is output quality critical?
              YES -> Reflexion (self-critique loop)
              NO  ->
                Is the problem open-ended/creative?
                  YES -> Tree-of-Thought
                  NO  ->
                    Unsure? -> Adaptive (auto-selects)
```

## Strategy Comparison

| Strategy | Best For | Avg Steps | Avg Tokens | Latency | Model Requirement |
|---|---|---|---|---|---|
| **ReAct** | Single/multi-step tool use, Q&A | 3–10 | 2K–8K | Fast | Any (4B+) |
| **Plan-Execute-Reflect** | Structured multi-step tasks with dependencies | 8–20 | 5K–15K | Medium | 8B+ recommended |
| **Tree-of-Thought** | Creative/open-ended exploration, brainstorming | 10–30 | 8K–25K | Slow | 14B+ recommended |
| **Reflexion** | Quality-critical output, self-correction | 6–15 | 4K–12K | Medium | 8B+ recommended |
| **Adaptive** | Mixed/unknown task types | Varies | Varies | Varies | 8B+ recommended |

## Strategy Details

### ReAct (Reactive)

The default strategy. Think-Act-Observe loop: the LLM thinks about what to do, calls a tool, observes the result, and repeats.

```typescript
.withReasoning({ defaultStrategy: "reactive" })
```

**Strengths:** Simple, fast, works with small models, low token cost.
**Weaknesses:** No planning ahead — can get stuck on complex multi-step tasks. Relies on the LLM to track progress implicitly.
**Loop detection:** Circuit breaker fires on repeated tool calls, repeated thoughts, or consecutive thoughts without action.

### Plan-Execute-Reflect

Creates a structured JSON plan first, then executes each step, then reflects and refines if needed.

```typescript
.withReasoning({
  defaultStrategy: "plan-execute-reflect",
  strategies: {
    planExecute: {
      maxRefinements: 2,
      reflectionDepth: "deep",
      stepRetries: 1,
    }
  }
})
```

**Strengths:** Handles complex multi-step tasks with dependencies. Plans are persisted to SQLite. Cross-step references (`{{from_step:sN}}`).
**Weaknesses:** Higher token cost from planning + reflection phases. Small models (<8B) struggle to produce valid JSON plans.
**Best with:** Cloud models (Claude, GPT-4) or large local models (14B+).

### Tree-of-Thought (ToT)

BFS exploration of multiple reasoning paths, scored and pruned. Best candidates are executed via ReAct.

```typescript
.withReasoning({
  defaultStrategy: "tree-of-thought",
  strategies: {
    treeOfThought: {
      breadth: 3,    // paths to explore per level
      depth: 3,      // levels of exploration
      pruningThreshold: 0.5,
    }
  }
})
```

**Strengths:** Explores creative alternatives. Finds non-obvious solutions. Self-scoring prevents bad paths.
**Weaknesses:** Highest token cost (breadth x depth LLM calls for exploration alone). Very slow on local models.
**Best with:** Cloud models only for practical use.

### Reflexion

Execute-then-critique loop. Runs the task, self-critiques the result, and retries with critique feedback.

```typescript
.withReasoning({
  defaultStrategy: "reflexion",
  strategies: {
    reflexion: {
      maxRetries: 3,
      selfCritiqueDepth: "deep",
    }
  }
})
```

**Strengths:** Improves output quality iteratively. Cross-run learning (prior critiques stored in episodic memory). Good for tasks where "good enough" is not enough.
**Weaknesses:** Multiplies execution time by retry count. Requires the model to be good at self-critique.
**Best with:** 8B+ models that can reason about their own output.

### Adaptive

Meta-strategy that analyzes the task and selects the best strategy automatically.

```typescript
.withReasoning({
  defaultStrategy: "adaptive",
  adaptive: { enabled: true, learning: true },
})
```

**Strengths:** No manual strategy selection needed. Learns from past executions when `learning: true`.
**Weaknesses:** Adds one LLM call for strategy selection. May not always pick optimally.
**Best with:** 8B+ models (needs to classify task complexity).

## Local Model Recommendations

| Model Size | Recommended Strategies | Avoid |
|---|---|---|
| 4B (qwen3:4b, phi-4-mini) | ReAct only | Plan-Execute, ToT, Adaptive |
| 8B (llama3.1:8b, qwen3:8b) | ReAct, Reflexion | ToT |
| 14B (cogito:14b, qwen3:14b) | ReAct, Plan-Execute, Reflexion | ToT (slow) |
| 70B+ (llama3.1:70b) | All strategies | None |
| Cloud (Claude, GPT-4, Gemini) | All strategies | None |

```typescript
/**
 * strategy-evaluator.ts — Structured LLM call to evaluate whether to switch
 * reasoning strategies when the current strategy gets stuck.
 *
 * Called by kernel-runner.ts when loop detection fires. Returns a decision
 * on whether to switch and which strategy to try next.
 */
import { Effect } from "effect";
import type { LLMService } from "@reactive-agents/llm-provider";
import { LLMService as LLMServiceTag } from "@reactive-agents/llm-provider";

// ── Types ────────────────────────────────────────────────────────────────────

/** State carried between strategies when a switch occurs. */
export interface StrategyHandoff {
  readonly originalTask: string;
  readonly previousStrategy: string;
  readonly stepsCompleted: number;
  readonly toolsCalled: string[];
  readonly keyObservations: string[];
  readonly failureReason: string;
  readonly switchNumber: number;
}

/** Input to the strategy evaluator. */
export interface StrategyEvalInput {
  readonly task: string;
  readonly currentStrategy: string;
  readonly stepsCompleted: number;
  readonly toolsCalled: string[];
  readonly failureReason: string;
  readonly availableStrategies: string[];
  readonly triedStrategies: string[];
}

/** Output from the strategy evaluator. */
export interface StrategyEvalResult {
  readonly shouldSwitch: boolean;
  readonly recommendedStrategy: string;
  readonly reasoning: string;
}

// ── Evaluator ────────────────────────────────────────────────────────────────

/**
 * Ask the LLM whether to switch strategies given the current stuck state.
 *
 * Uses a small structured output call. The prompt includes:
 * - Task description
 * - Current strategy and why it's stuck
 * - Available alternatives (excluding already-tried)
 * - Instructions to output JSON: { shouldSwitch, recommendedStrategy, reasoning }
 */
export function evaluateStrategySwitch(
  input: StrategyEvalInput,
): Effect.Effect<StrategyEvalResult, never, LLMService> {
  return Effect.gen(function* () {
    const llm = yield* LLMServiceTag;

    // Filter out already-tried strategies and current strategy
    const candidates = input.availableStrategies.filter(
      (s) => s !== input.currentStrategy && !input.triedStrategies.includes(s),
    );

    // If no candidates, don't switch
    if (candidates.length === 0) {
      return {
        shouldSwitch: false,
        recommendedStrategy: "",
        reasoning: "No alternative strategies available — all have been tried",
      };
    }

    const systemPrompt = `You are a reasoning strategy advisor. Given a stuck agent, decide whether to switch to a different reasoning strategy.

Available strategies:
${candidates.map((s) => `- ${s}`).join("\n")}

Strategy descriptions:
- reactive: Think-Act-Observe loop. Best for simple tool-use tasks.
- plan-execute-reflect: Create structured plan, execute steps, reflect. Best for complex multi-step tasks.
- tree-of-thought: BFS exploration of multiple reasoning paths. Best for creative/open-ended problems.
- reflexion: Execute-critique-retry loop. Best for quality-critical output.
- adaptive: Auto-selects strategy based on task analysis.

Respond with ONLY valid JSON: { "shouldSwitch": boolean, "recommendedStrategy": string, "reasoning": string }
If shouldSwitch is false, set recommendedStrategy to "".`;

    const userPrompt = `Task: ${input.task}

Current strategy: ${input.currentStrategy}
Steps completed: ${input.stepsCompleted}
Tools called: ${input.toolsCalled.join(", ") || "none"}
Failure reason: ${input.failureReason}

Should we switch to a different strategy? If yes, which one?`;

    const noSwitch: StrategyEvalResult = {
      shouldSwitch: false,
      recommendedStrategy: "",
      reasoning: "LLM evaluation failed — defaulting to no switch",
    };

    return yield* llm.complete({
      messages: [
        { role: "system" as const, content: systemPrompt },
        { role: "user" as const, content: userPrompt },
      ],
      temperature: 0.3,
      maxTokens: 200,
    }).pipe(
      Effect.map((response) => {
        // Parse structured output with fallback
        try {
          const content = response.content.trim();
          // Extract JSON from potential markdown code blocks
          const jsonMatch = content.match(/\{[\s\S]*\}/);
          if (jsonMatch) {
            const parsed = JSON.parse(jsonMatch[0]);
            return {
              shouldSwitch: Boolean(parsed.shouldSwitch),
              recommendedStrategy: String(parsed.recommendedStrategy ?? ""),
              reasoning: String(parsed.reasoning ?? ""),
            };
          }
        } catch {
          // Parse failure — don't switch
        }
        return noSwitch;
      }),
      // Catch LLM failures to maintain the `never` error channel
      Effect.catchAll(() => Effect.succeed(noSwitch)),
    );
  });
}

/**
 * Build a handoff context string from a StrategyHandoff.
 * Injected as system context prefix for the new strategy.
 */
export function formatHandoffContext(handoff: StrategyHandoff): string {
  const lines = [
    `[Strategy Switch ${handoff.switchNumber}]`,
    `Previous approach: ${handoff.previousStrategy} (${handoff.stepsCompleted} steps)`,
    `Failure reason: ${handoff.failureReason}`,
  ];
  if (handoff.toolsCalled.length > 0) {
    lines.push(`Tools already called: ${handoff.toolsCalled.join(", ")}`);
  }
  if (handoff.keyObservations.length > 0) {
    lines.push(`Key observations so far:`);
    for (const obs of handoff.keyObservations) {
      lines.push(`  - ${obs}`);
    }
  }
  lines.push(`Now trying: different strategy. Build on the observations above.`);
  return lines.join("\n");
}
```

**Important note on architecture:** The kernel runner itself does NOT re-dispatch to a new strategy. It returns a "failed" state with `meta.strategySwitch` populated. The **calling strategy function** (in `reactive.ts`, etc.) or the **execution engine** reads this meta and re-invokes with the new strategy. This keeps the kernel runner simple and avoids circular dependencies.
