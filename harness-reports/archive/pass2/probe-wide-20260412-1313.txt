1. **B-tree Indexing**  
   - **Structure**: A balanced tree structure that maintains sorted data, enabling efficient lookups, inserts, and deletes.  
   - **Use Cases**: Ideal for range queries (e.g., `WHERE column > value`), sorted data retrieval, and scenarios requiring ordered results.  
   - **Performance Trade-offs**: Efficient for equality, range, and ordered queries but slower than hash indexes for exact matches. Scales well for large datasets but consumes more memory due to its tree hierarchy.  

2. **Hash Indexing**  
   - **Structure**: Uses a hash function to map keys directly to specific locations (buckets), enabling rapid exact-match lookups.  
   - **Use Cases**: Best suited for equality queries (e.g., `WHERE column = value`).  
   - **Performance Trade-offs**: Extremely fast for exact matches but inefficient for range queries or sorted data. Susceptible to collisions and less scalable for large datasets with high cardinality.  

3. **Full-Text Indexing**  
   - **Structure**: Designed for text data, using inverted indexes to store term-to-document mappings, supporting partial matches, wildcards, and natural language queries.  
   - **Use Cases**: Optimized for searching unstructured or semi-structured text (e.g., `WHERE column LIKE '%keyword%'`).  
   - **Performance Trade-offs**: Excels in text searches but performs poorly for exact matches or numerical ranges. Requires significant storage and computational resources due to its complexity.  

**When to Use Each**:  
- Use **B-trees** for general-purpose indexing, especially with range queries or ordered data.  
- Use **hash indexes** for high-speed equality checks on low-cardinality or static data.  
- Use **full-text indexes** for complex text searches, such as in search engines or document repositories.
