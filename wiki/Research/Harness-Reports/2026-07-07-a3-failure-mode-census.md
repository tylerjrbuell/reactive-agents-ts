## Honesty-tracer / reliability note

Per the methodology caveat above, no `dishonest-success-suspected` / `claimed-success (unverified)` strings exist in these trace files to check against directly. Reconstructing the closest available proxy — **"status:success" (or a terminal guard reason) with zero substantive tool calls on a `requiresTools:true` task** — surfaces exactly 2 cases in the 20 ra-full traces: #4 (rw-8 all-analysis plan) and #6 (rw-2 cogito, though this one correctly ships `status: failure`, not a false success). That 1-in-20 rate is a lower bound — it can't see cases where a tool *was* called but did nothing to ground the actual claim (the cogito report's rw-1/rw-9 coin-flip pattern), because that requires the judge's semantic read, not raw event presence.

## Priority ranking for follow-up (frequency × severity × novelty)

1. **#3 (compressed-preview-in-tool-args)** — highest priority. Deterministic, 100%-reproducible on any plan-execute chained-search task, directly implicated in rw-1's headline −67% lift, and entirely unaddressed by any shipped or planned fix. Single-function fix (`resolveStepReferences`).
2. **#4 (all-analysis plan ungrounded success)** — second priority. Rare in this corpus (n=1) but catastrophic when it fires (100% false-positive success, zero runtime detection) and structurally guaranteed to recur on any `requiresTools` task where the planner chooses an all-analysis decomposition.
3. **#1 (B2, thinking-token thrash in the planning sub-kernel)** — already tracked as B2 for the react loop; this census adds the plan-execute planner as a second, unaddressed call site for the same fix.
4. **#6 (abstention vs. loop-detector race)** — lower frequency (n=1) but cheap to fix and directly improves the honesty of failure reporting (abstained > failure as a terminal signal).
5. **#5, #9, #11, #12** — lower priority, mostly hardening/observability, bundle with whichever of the above lands first.
