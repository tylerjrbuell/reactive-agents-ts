# Leading Harness Practices vs. Reactive Agents (2026-07-07)

**Scope:** harness/runtime mechanics only — context management, tool-result shaping, termination discipline, sub-agent delegation, retry/reliability primitives, local-model tool-call tricks, and runtime verification/grounding. Explicitly excludes model quality and UI.

**Method:** WebSearch/WebFetch against current (mid-2026) docs/blogs/repos for Claude Code, Codex CLI, Gemini CLI, LangGraph, Mastra, OpenAI Agents SDK, Vercel AI SDK, Ollama/llama.cpp, and harness-engineering literature; cross-checked against `packages/reasoning/src/kernel`, `packages/reasoning/src/context`, `packages/reasoning/src/assembly`, `packages/llm-provider/src`, `packages/runtime/src/builder/build-effect`, and `packages/tools/src` in this repo.

## 1. Practices Table

### A. Context economy / compaction / tool-result shaping

| # | Harness | Mechanism (concrete) | Source | RA status |
|---|---|---|---|---|
| A1 | Claude Code | Auto-compact triggers at ~95% of context window: pauses the turn, summarizes full history, replaces earlier turns with the summary, keeps current file states/task objective/recent tool results, discards rejected approaches and verbose intermediate output. | [Claude Code Context Buffer](https://claudefa.st/blog/guide/mechanics/context-buffer-management), [Compaction docs](https://platform.claude.com/docs/en/build-with-claude/compaction) | **HAS** — `packages/reasoning/src/context/message-window.ts::applyMessageWindowWithCompact` fires at 75% of `maxTokens`, keeps first user message + tier-adaptive N recent turns in full, folds older turns into `[Prior: called X → brief]` one-line summaries. Threshold is lower (75% vs 95%) and per-turn-group aware. |
| A2 | Codex CLI | Compacts by replacing the input list with a smaller "representative" item list once `auto_compact_limit` is exceeded, via a dedicated compaction endpoint. | [Unrolling the Codex agent loop](https://openai.com/index/unrolling-the-codex-agent-loop/) | **HAS** (same mechanism family) — `packages/reasoning/src/assembly/stages/compact-history.ts` does a char-budget check (`window*4`) and half-history summarization fallback; `message-window.ts` is the primary path. Two overlapping compaction stages exist (see gap note below). |
| A3 | Gemini CLI | `ChatCompressionService`: threshold defaults to 50% of context (not 95%), summarizes older turns, protects `GEMINI.md`/`/memory add` content verbatim as "authoritative reference material," tracks `CompressionStatus` (COMPRESSED / COMPRESSION_FAILED_INFLATED_TOKEN_COUNT / CONTENT_TRUNCATED) so the harness knows when a compaction pass made things worse. | [DeepWiki: Chat Compression](https://deepwiki.com/google-gemini/gemini-cli/4.12-chat-compression-and-context-management), [Gemini CLI compression changelog](https://www.yet-another-changelog.ai/changelog/gemini-cli-automatic-context-compression-service) | **PARTIAL** — RA protects instructions similarly (`preservedOldUsers` in `message-window.ts` keeps mid-thread user messages verbatim instead of folding them). **Missing:** no outcome-tracking enum (RA doesn't detect "the compaction summary came out *longer* than what it replaced" — a documented Gemini CLI failure mode with its own status code) and no dedicated "protected/authoritative" system-prompt-level content class analogous to GEMINI.md. |
| A4 | Vercel AI SDK 6 | `prepareStep({stepNumber, messages, steps, model}) → mutate` runs before every loop iteration; ships a `pruneMessages()` helper (`reasoning:'all'`, `toolCalls:'before-last-3-messages'`, `emptyMessages:'remove'`) plus per-step model swapping and `activeTools` restriction by phase. | [AI SDK Loop Control](https://ai-sdk.dev/docs/agents/loop-control) | **PARTIAL** — RA's per-tier `toolSchemaDetail` (`names-only`/`names-and-types`/`full`) and `tool-gating.ts` narrow the active toolset per-decide-cycle, and `applyCapabilityMaxTokens` swaps context budget by resolved capability — but there's no single generic "run this hook before every step to mutate state arbitrarily" extension point; the equivalent logic is hard-coded across kernel capability modules rather than exposed as one composable step-hook. |
| A5 | Claude Code | Bash tool output truncated (~2KB/30K chars docs vary); large tool outputs generally truncated before hitting context; open GitHub issue (#40100) asks for truncation to scale with context window since flat caps under-serve 1M-token models. | [Bash tool docs](https://platform.claude.com/docs/en/agents-and-tools/tool-use/bash-tool), [GH #40100](https://github.com/anthropics/claude-code/issues/40100) | **HAS, and arguably ahead** — `packages/reasoning/src/context/context-profile.ts` already scales `toolResultMaxChars`/`toolResultPreviewItems` *by tier* (4000 chars local / 1200 mid / 800 large / 600 frontier) rather than a flat cap — the exact fix Claude Code's own backlog is asking for. |
| A6 | Codex/Claude Code/harness-eng literature | Verbatim compression (50–70% reduction with zero paraphrase drift) beats larger raw windows; "compact budget under 30K tokens + high info density" outperforms diluted large-context stuffing. | [Externalization in LLM Agents survey](https://arxiv.org/pdf/2604.08224) | **HAS (partial credit)** — tiered `toolResultMaxChars` + preview-item truncation is a density strategy; RA's `[Prior: ...]` fold is lossy summarization, not verbatim excerpting, so it doesn't fully match the "verbatim compression" ideal cited in the survey. |

### B. Termination discipline / plan tracking / self-correction

| # | Harness | Mechanism | Source | RA status |
|---|---|---|---|---|
| B1 | Vercel AI SDK / OpenAI Agents SDK | Declarative stop conditions: `isStepCount(20)` default safety cap, `hasToolCall(name)`, custom `({steps}) => boolean`; OpenAI Agents SDK adds max-turn limits + guardrail-exception-triggered termination as circuit breakers, with full run tracing (model calls, tool calls, handoffs, guardrail outcomes) for post-hoc diagnosis. | [AI SDK Loop Control](https://ai-sdk.dev/docs/agents/loop-control), [OpenAI Agents SDK 2026 guide](https://callsphere.ai/blog/openai-agents-sdk-2026-multi-agent-systems-handoffs-guardrails) | **HAS** — `packages/reasoning/src/kernel/loop/terminate.ts` is the documented single termination owner; `maxIterations` per tier (8/10/10/12) is the step-count cap; the Arbitrator (`decide/arbitrator.ts`) is the equivalent of a custom stop-predicate consuming step history. |
| B2 | Gemini CLI | `LoopDetectionService`: checks for repeating tool-call/thought patterns before each turn AND mid-stream per event; on first detection attempts recovery, on repeat aborts with a user-facing choice to disable detection for the session. | [Read OSS: Inside the Agentic Loop](https://readoss.com/en/google-gemini/gemini-cli/inside-agentic-loop-how-gemini-cli-processes-prompt), GH issue #8928 | **HAS** — `packages/reasoning/src/kernel/capabilities/reflect/loop-detector.ts` detects (a) repeated identical tool+args, (b) repeated thought content, (c) consecutive-thoughts-without-action, and routes the verdict through the same single Arbitrator rather than a separate loop-abort code path — arguably cleaner than Gemini CLI's bolt-on service. |
| B3 | Claude Code | `TodoWrite` tool: required for any 3+-step task, marked `in_progress` before starting, **"ONLY mark a task as completed when you have FULLY accomplished it"** — explicitly forbidden to mark complete if tests fail/errors remain/work is partial; exactly one `in_progress` task at a time; user-visible throughout the run. | [Piebald-AI system prompts repo — TodoWrite](https://github.com/Piebald-AI/claude-code-system-prompts/blob/main/system-prompts/tool-description-todowrite.md) | **PARTIAL / MISSING** — `packages/reasoning/src/strategies/plan-execute.ts` has a strongly-typed internal `Plan`/`PlanStep` (via `hydratePlan`, `computeWaves`) with per-step retry-then-LLM-patch-then-reflect, but this artifact (a) only exists inside the `plan-execute` strategy, not exposed as a generic cross-strategy tool any strategy (reactive, reflexion, etc.) can call, and (b) is not surfaced to the caller as a live, incrementally-updated user-visible todo list the way `TodoWrite` is. There is no `todo`-tool anywhere under `packages/tools/src` (confirmed via grep — zero hits). This is the single most direct "Claude Code has it, RA doesn't" gap. |
| B4 | harness-engineering lit (maker/checker pattern) | **"Never let the AI verify its own done. Maker and checker must be different things."** Layered gates (syntax → runtime/test execution → end-to-end) each must pass sequentially; an independent, deliberately "nitpicky" evaluator agent catches what self-evaluation misses because self-eval shares the same blind spots as the generation step. | [Lecture 9: Preventing Agents from Declaring Victory Too Early](https://walkinglabs.github.io/learn-harness-engineering/en/lectures/lecture-09-why-agents-declare-victory-too-early/), [Harness Engineering: Maker/Checker Split](https://rickhigh.substack.com/p/loop-engineering-the-makerchecker) | **PARTIAL** — RA's `packages/reasoning/src/kernel/capabilities/verify/verifier.ts` + `evidence-grounding.ts` do deterministic, non-LLM, pure-function checks (numeric-claim-in-evidence-corpus grounding, fabricated-measurement detection, scaffold-leak detection) which *are* architecturally separate from the generating call — good. But the LLM-side quality check (`critique.ts`, used by reflexion + plan-execute-reflect) is explicitly **self**-critique: same agent, same conversation, asked to grade its own output. There is no default wiring of a separate model/tier as an independent checker in the live run loop. `packages/judge-server` implements exactly this pattern (separate judge process/model) but is wired only into the offline eval harness (`packages/eval`), not into `packages/runtime`'s live execution path — confirmed via grep, zero references from `packages/reasoning` or `packages/runtime` to judge-server. |
| B5 | OpenAI Codex CLI / Claude Code | `/plan`, `/exec`, `/review` command triad and multi-hour autonomous sessions rely on explicit plan-then-execute staging with checkpoints. | [Codex CLI docs](https://developers.openai.com/codex/cli) | **HAS** (as a strategy, not a universal primitive) — `blueprint`/`plan-execute` strategies cover this; `strategy-registry.ts` lets a run pick it. Gap is the same as B3: not universal across strategies. |

### C. Sub-agent delegation

| # | Harness | Mechanism | Source | RA status |
|---|---|---|---|---|
| C1 | Claude Code | Subagents run in their own context window with custom system prompt + scoped tools; only the subagent's *final message* returns to the parent — all intermediate tool calls/reads stay inside the subagent's context, never inflating the parent's. Description-field-driven automatic delegation. "Dynamic Workflows" (May 2026) fan a lead agent's work across up to 1,000 subagents. | [Create custom subagents](https://code.claude.com/docs/en/sub-agents), [Claude Code Subagents guide](https://medium.com/@sathishkraju/claude-code-subagents-the-complete-guide-to-ai-agent-delegation-d0a9aba419d0) | **HAS** — `packages/runtime/src/builder/build-effect/sub-agent-executor.ts` builds an isolated sub-runtime via `createLightRuntime`, filters tools (`META_TOOL_NAMES` stripped from delegated-tools-used), composes a scoped persona/system prompt (`buildSubAgentSystemPrompt`), and returns only the unwrapped `TaskResult` to the parent. `spawn-handlers.ts` provides both single (`spawn-agent`) and batch (`spawn-agents`) fan-out — the batch path is RA's analog to Dynamic Workflows fan-out, though at much smaller documented scale. |

### D. Reliability primitives (retry, memory/session, structured tool calls)

| # | Harness | Mechanism | Source | RA status |
|---|---|---|---|---|

| D2 | LangGraph | Checkpointer persists full graph state after every node; resume from last checkpoint on crash rather than restart from node 1; explicit warning that checkpointing has **no built-in failure-detection/watchdog** — a process crash is silent unless the caller adds one. | [LangGraph Persistence docs](https://docs.langchain.com/oss/python/langgraph/persistence), [Diagrid: Checkpoints ≠ Durable Execution](https://www.diagrid.io/blog/checkpoints-are-not-durable-execution-why-langgraph-crewai-google-adk-and-others-fall-short-for-production-agent-workflows) | **HAS** — per project memory, durable crash-resume shipped 2026-06-12 (`run_approvals` table, checkpoint-based resume); worth independently confirming whether RA has closed LangGraph's documented watchdog gap (out of scope for this pass — flagged, not verified). |
| D3 | All 6 providers in llm-provider (this repo) / industry-standard | Exponential backoff retry restricted to rate-limit/timeout errors; circuit breaker with failure threshold + cooldown + half-open probe. | n/a (internal) | **HAS** — `packages/llm-provider/src/retry.ts`: `Schedule.exponential("1 second", 2.0)` capped at 3 retries, gated to `LLMRateLimitError`/`LLMTimeoutError` only; `circuit-breaker.ts` implements `failureThreshold:5, cooldownMs:30_000, halfOpenRequests:1`. | D4 | Ollama | Since 0.3.0, `format` accepts a JSON schema and constrains decoding via XGrammar (grammar-constrained), giving structural (not semantic) 100% schema compliance; production guidance is to treat it as "a strong hint + app-level validation," not a trusted contract. | [Ollama JSON schema issue #6002](https://github.com/ollama/ollama/issues/6002), [Constraint Tax paper](https://arxiv.org/pdf/2606.25605) | **HAS** — `packages/llm-provider/src/providers/local.ts` (~line 844) builds an "Ollama-native format constraint... for schema-enforced JSON output (GBNF grammar under the hood)" for structured output extraction — same XGrammar/GBNF mechanism, same caveat structure (schema conformity ≠ semantic correctness) as the ecosystem-standard warning. |
| D5 | llama.cpp | GBNF grammars convert JSON-schema tool definitions into token-level accept/reject masks during sampling (`llama_grammar_apply_impl`), forcing every generated token to be grammar-valid — the mechanism Ollama and most local tool-calling stacks build on. | [llama.cpp grammar docs](https://mintlify.com/ggml-org/llama.cpp/advanced/grammars), [DeepWiki: Grammar and Structured Output](https://deepwiki.com/ggml-org/llama.cpp/8.1-grammar-and-structured-output) | **HAS (inherited via Ollama)** — RA doesn't re-implement grammar sampling itself; it depends on Ollama's XGrammar layer for local-model structured output, which is the standard integration point (frontier providers use their own native structured-output/tool-call APIs instead — also wired, per project memory `.withOutputSchema`). |
| D6 | harness-eng lit | Thinking-token budget starvation: on reasoning-mode models, `num_predict`/output budget must be widened because chain-of-thought tokens eat into the same cap as the visible answer, or the model empty-content-times-out. | (general local-model harness pattern; corroborated by search result on context-window/thinking-budget tricks) | **HAS** — `packages/reasoning/src/context/context-profile.ts` `thinkingModel: Schema.optional(Schema.Boolean)` field, set via `applyCapabilityMaxTokens` when `capability.supportsThinkingMode`, exists specifically "the kernel widens its per-turn output budget for these models: thinking tokens count against num_predict, so a flat tier cap starves the visible answer" — this is a direct, recently-added (2026-07-07, per in-code comment "B2") fix for exactly this documented local-model failure mode. |

### E. System-prompt / tool-schema economy

| # | Harness | Mechanism | Source | RA status |
|---|---|---|---|---|
| E1 | Claude Code | System prompt ~2.5K tokens; tool definitions 14–17K tokens (the dominant cost); tool descriptions are dense-but-not-minimal — e.g. the Bash tool costs 1,558 tokens because it embeds git-commit/PR-formatting instructions that pay for themselves in output quality. Neither is strictly better; RA's approach is adaptive per model tier, Claude Code's is adaptive per human review of what's "load-bearing." |

## 2. MISSING-ranked shortlist (highest-leverage first)

Ranked by how directly each explains "a lean manual loop can beat a rich harness on a strong local model" — i.e., practices that reduce wasted context/tokens or prevent false-positive termination, which is exactly where a hand-rolled loop has an advantage (nothing to misfire).

1. **No universal, cross-strategy, user-visible plan/todo tracker (B3).** Claude Code's `TodoWrite` enforces "only mark complete when fully done, keep exactly one in-progress" as a structural constraint every strategy obeys. RA has this logic *only* inside `plan-execute.ts` (typed `Plan`/`PlanStep`), invisible to `reactive`/`reflexion`/other strategies and not exposed as a callable tool. This is the cleanest, most surgical gap to close — likely a new tool in `packages/tools/src` plus kernel wiring, not a redesign.

2. **No independent (different-model/different-process) checker in the live run loop (B4).** `packages/judge-server` already implements exactly this pattern but only feeds `packages/eval` (offline benchmarking), not `packages/runtime`'s live execution path. The deterministic evidence-grounding checks in `verifier.ts` are good and separate-by-construction, but the LLM-quality half of verification (`critique.ts`) is same-model self-grading — the single most-cited failure mode in the harness-engineering literature ("any blind spot in production is also a blind spot in evaluation").

3. **No compaction-outcome tracking (A3).** Gemini CLI's `CompressionStatus` enum (COMPRESSED / COMPRESSION_FAILED_INFLATED_TOKEN_COUNT / CONTENT_TRUNCATED) lets the harness detect when its own summarization pass backfired (output longer than input, or fell back to truncation). RA's `message-window.ts` and `compact-history.ts` have no equivalent self-check — a summary that expands instead of shrinking would silently pass through.

4. **Two overlapping compaction code paths, no single generic step-hook (A2/A4).** `compact-history.ts` (char-budget, `window*4`, half-history cut) and `message-window.ts` (token-estimate, 75% threshold, turn-aware fold) both exist and can plausibly both fire in the same run. Vercel AI SDK's `prepareStep` gives a single point where all step-boundary mutation (compaction, model-swap, tool-gating) composes explicitly; RA's equivalent logic is scattered across kernel capability modules. Low urgency but worth consolidating before it causes a double-compaction bug.

5. **No "protected/authoritative content" class during compaction (A3).** Gemini CLI explicitly shields `GEMINI.md` and `/memory add` entries from compression as user-curated, must-stay-verbatim facts. RA preserves ungrouped user messages (`preservedOldUsers`) but has no equivalent concept for system-level "never compact this" instructions distinct from ordinary conversation turns.

**Not gaps — RA already ahead or at parity:** tier-scaled tool-result truncation (A5, ahead of Claude Code's flat-cap backlog item), loop detection (B2, cleaner single-owner routing than Gemini CLI's bolt-on service), sub-agent delegation with isolated context (C1), retry/circuit-breaker (D3), Ollama grammar-constrained structured output (D4/D5), and thinking-token budget widening (D6, shipped same day as this report per in-repo comments).
