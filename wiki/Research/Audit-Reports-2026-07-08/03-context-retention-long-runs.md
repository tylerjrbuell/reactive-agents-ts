# Architecture Sweep 2026-07-08 — 03-context-retention-long-runs

Audited at `f4213c07` (post Phase-1b APC deletion `279b61fb`, post hotfixes 0.5-1 guidance / 0.5-2 `:full` / 0.5-4 semantic-full-store). Builds on 2026-07-07 reports 02 (prompts/assembly) and 06 (compression/grounding). All paths relative to `packages/reasoning/src/` unless noted.

## Findings (ranked by leverage)

**F1 — `priorContext` is WRITE-ONLY: strategy-switch handoffs, ToT planning handoffs, and memory context never reach the model.** The only renderer of `KernelInput.priorContext` was the dead APC stack — `context-manager.ts:224` + `prompt-sections-default.ts:115` (`priorContextSection`) at `279b61fb^` — which report 02 already established had *no live caller* and which Phase 1b then deleted. Today the field has writers everywhere and zero prompt readers:
- strategy-switch folds the 9-line handoff summary into it (`kernel/loop/runner-helpers/strategy-switch.ts:201-203`),
- tree-of-thought passes its selected-plan summary (`strategies/tree-of-thought.ts:690`),
- reactive/direct fence `memoryContext` into it (`strategies/reactive.ts:172-174`, `strategies/direct.ts:113-115`),
- react-kernel threads it through `buildKernelInput` (`kernel/loop/react-kernel.ts:223`; `kernel/state/build-kernel-input.ts:73`).
No file in `kernel/`, `assembly/`, `context/`, or `strategies/` concatenates it into a system prompt or message. `systemPromptStage` (`assembly/stages/system-prompt.ts`) and `project()` never see it. This is the exact same disease as report-02 F2 (guidance assembled-then-dropped) — that sibling was fixed by hotfix 0.5-1 (`context/guidance.ts:1-19`), this one was not. Consequence: **after a strategy switch the model restarts blind** (see P4 section), ToT's planning phase is wasted on the execution kernel, and reactive/direct memory bootstrap never reaches the prompt through this rail.

**F2 — The recall meta-tool round-trip is structurally dead under canonical assembly: full stored evidence is model-unreachable for re-reading.** The chain:
1. Execution-side compression stores the FULL result in the scratchpad under `_tool_result_N` and appends a hint to the message content: `— full text is stored. Use recall("<key>") to retrieve.` (`kernel/capabilities/act/tool-execution.ts:733-735`; also `act/conversation-assembly.ts:112-119` for the >4000-char inline-cap form).
2. But the next think turn does NOT send `state.messages` content. `fromKernelState` (`assembly/from-kernel-state.ts:105-121`) maps each `tool_result` message with a `storedKey` to its scratchpad ref, and `projectResultsStage` re-renders content from the ResultStore: `store.materialize(ref)` (full) or `store.preview(ref, budget)` (`assembly/stages/project-results.ts:79-99`). The preview footer speaks a *different retrieval vocabulary*: `result_ref="…" … write_result_to_file(result_ref=…)` (`assembly/result-store.ts:79-83`) — by design "no `[STORED:]` marker, no `recall(` hint" (project-results.ts:17-18). The recall hint written at execution time is systematically overwritten at render. `grep 'recall("' assembly/` = zero hits.
3. The recall-overflow gate hides the `recall` tool unless a `recall("` marker is visible in the CURRENT projected window: `SURFACED_RECALL_KEY = /recall\("[^"]+"/` tested against `conversationMessages` = `toLLMMessages(request.messages)`, i.e. the post-projection thread (`kernel/capabilities/reason/think-guards.ts:544-561`; `think.ts:432,469-472`). Since the projection never emits `recall("`, the gate never opens (only escape: calibration `observationHandling === "uses-recall"` force-on, think.ts:469, or `RA_RECALL_GATE=0`).
4. The surviving affordance, `write_result_to_file`, returns `{written, path, ref, bytes, items}` — it writes the full data to DISK and returns metadata, not content (`packages/tools/src/skills/write-result-to-file.ts:47,117`). Re-reading into context is a 2-hop `write → read-file` dance that nothing in the prompt suggests and that requires a file-read tool on the surface.

Net: the "5000-token search result becomes a 200-char preview, full version retrievable at iteration 25" story is **half true — the full version is retained system-side forever (scratchpad is never evicted: no delete/clear/prune sites outside the `_todo` reset, and it is checkpoint-serialized at `kernel/state/kernel-state.ts:1031/1061`) but the model has no working tool path to read it back**. The gate's own doc comment (think-guards.ts:528-530, "message-window compaction leaves no recall pointer") describes the pre-canonical world; the deleted `message-window.ts` is still referenced in stale comments at `conversation-assembly.ts:104`, `reactive-observer.ts:400`, `kernel-state.ts:274`.

**F3 — The dominant evidence-loss event is NOT window aging — it is the preview cut ONE iteration after gathering.** Assembly math (see kernel-state section): with per-result preserve budgets of 4000/1200/800/600 chars (local/mid/large/frontier, `context/context-profile.ts:81/98/107/115`, single-sourced via `assembly/capability.ts:46-51`), a result larger than the budget is truncated to a heading-skeleton/head preview *from the very next iteration onward* — only the LATEST result gets the generous recency budget (`window×0.35×4` chars ≈ 45.9k on a 32k window, capability.ts:63; project-results.ts:86-88). So a fact sitting at char 3000 of a 5000-char search result gathered at iteration 3 is invisible at iteration 4, not iteration 25 — and F2 means it is unrecoverable by the model thereafter. Meanwhile `compactHistoryStage` (the "aging out" path) fires only when total thread chars exceed `window×4` = 131k chars on a 32k window (`assembly/stages/compact-history.ts:5-7`) — at ≤1200 chars/result on mid that's ~100 tool results, i.e. essentially never at 30 iterations. Retention policy is therefore effectively: *latest result rich, everything else skeletal, forever*.

**F4 — Compaction, when it does fire, destroys refs and pretends to summarize.** `compactHistoryStage` drops the older HALF of messages and inserts `[history compacted: N earlier messages summarized]` (compact-history.ts:16-17) — nothing is summarized, no `result_ref`s or keys are carried into the stub, and the goal user-turn (messages[0]) is in the dropped half (goal survives only via the system-prompt `Goal:` line, `system-prompt.ts`). It is single-pass (no re-check that the kept half fits) and has **no protected classes**: the `preserveOnCompaction` flag is computed and stored on every observation (`kernel/utils/observation-helpers.ts:32,43`) but read by NOTHING — a dead protected-class rail. There is no post-compaction "is the answer still derivable" check (confirms report-06 finding 3; the census there shrinks by one — path #1 `applyMessageWindowWithCompact` was deleted with the APC stack, leaving 5 uncoordinated paths: render-side compactHistory, execution-side compressToolResult, context-pressure tool narrowing, from_step caps, truncateForDisplay).

**F5 — Strategy switch (P4): the harness remembers, the model does not.** `applyStrategySwitch` carries: (a) the last 8 successful observation STEPS + `toolsUsed` set (`strategy-switch.ts:156-169`), (b) synthetic failed-tool steps (:174-195), (c) the handoff summary into `priorContext` (:201-203). But: carried steps live in `state.steps[]`, which `project()` NEVER renders — the assembly reads only `state.messages` + scratchpad (`from-kernel-state.ts`); `initialKernelState` starts the new strategy with an EMPTY message thread; and the handoff summary goes into the write-only `priorContext` (F1). So the carryover benefits only steps-based gates (grounded-terminal, required-tools redirect — the "don't re-call succeeded tools" fix) while the **LLM window after a switch contains exactly: system prompt + goal.** Also lost on switch: plan-execute plan state (in-strategy local), `pendingGuidance`, and `state.scratchpad` (reset with fresh state — though the process-global `scratchpadStoreRef` still holds all entries and act's end-of-round merge restores them into state at the next tool round, `act.ts:924-929`; model-unreachable either way per F2). Survives switch: todo list (shared Ref, keyed `_todo:<taskId>`), toolsUsed, ≤8 steps for gates.

**F6 — No dedup/cache prevents the re-read tax.** When evidence is invisible (preview-cut or compacted), the only mechanisms around re-gathering are: loop detection of *consecutive* identical actions (same tool+args N in a row, `kernel/capabilities/reflect/loop-detector.ts:44-60`) and the post-switch required-tools floor using carried `toolsUsed`. A duplicate call at iteration 25 of iteration-3's search — separated by other actions — is executed again at full cost: no tool-result cache, no content-hash dedup (ResultStore's `put()` content-hashing is per-render and system-side only), no "you already ran this, key=X" nudge. Nothing even detects it for telemetry.

**F7 — Within-run memory is write-only; memory helps only ACROSS runs.** Semantic stores happen fire-and-forget per successful tool call, now with full content (hotfix 0.5-4, `tool-execution.ts:606-620,743-760`). But in-run retrieval: `RecallService` is a Phase-1 seam with `NoopRecallServiceLayer` default; even when a layer IS provided, `iterate-pass.ts:445-470` captures `iterRecallContext`/`iterRecallSkills` and immediately `void`s them ("Phase 1 leaves … unused beyond capture"). The `find` tool's `memory` scope greps only `state.bootstrapMemoryContent` — the run-START snapshot (`packages/tools/src/skills/find.ts:105-118`), never live writes. So a fact stored to semantic memory at iteration 3 is not retrievable by any means at iteration 25 of the same run; it becomes useful only to the NEXT run's bootstrap.

## Target-by-target detail

### context/ + assembly (what enters the window; eviction order)
Live pipeline is `project()` = systemPrompt → selectTools → projectResults → compactHistory → finalize (`assembly/project.ts:39-45`), invoked per-iteration from `buildThinkProviderRequest` (`think.ts:176-190`) via `fromKernelState`. It is a **stateless re-projection**: every iteration rebuilds the whole thread from `state.messages` + scratchpad, so "eviction" is a per-render policy, not state mutation. Eviction order per iteration: (1) every non-latest tool result capped at `toolResultPreserveBudget` → structure-aware preview+`result_ref` footer (project-results.ts:86-99, result-store.ts:73-110); (2) if total > `window×4` chars, drop older half with a no-ref stub (compact-history.ts). Guidance signals now reach the prompt via the dynamic tail (hotfix 0.5-1, `context/guidance.ts`). `state.messages` and `state.steps` themselves are never trimmed — unbounded system-side growth, render-side capping only.

### Compression + scratchpad round-trip
Store side works: `compressToolResult` → preview + `_tool_result_N` full value in `scratchpadStoreRef` (tool-execution.ts:548-564 text path, :706-736 native-FC path), merged into `KernelState.scratchpad` each act round (act.ts:924-929), checkpoint-serialized, never evicted. `extractedFact` (deterministic, pre-compression, tool-execution.ts:691-704) is stored per-step and feeds the evidence-grounding corpus (`think-guards.ts:485-493`) and switch-handoff keyObservations — but is never rendered into the model window. Retrieval side is broken (F2). Sub-budget results get no storedKey; they're inline until compaction, then gone (fromKernelState mints render-only content-hash refs for them, from-kernel-state.ts:111-114, but no key ever surfaces to the model).

### kernel-state growth over 30 iterations (the math)
Mid tier defaults: window 32,768 tokens (`context-profile.ts:97-103` maxTokens) → compact threshold 131,072 chars; per-result preserve 1,200 chars; recency budget ≈45,875 chars. A 30-iteration run with one tool call each ≈ goal + 30×(assistant 0-char turn + ≤1,200-char result) ≈ 36k chars → compaction never fires; **iteration-1 evidence is still "visible" at iteration 30 — but only its first ≤1,200 chars (or heading skeleton)**. The visibility cliff is the preview cut at iteration 2 (F3), not window aging. Local tier: 4,000×30 = 120k chars — brushes the 131k limit around iteration ~30 with chatty tools, at which point the older half (iterations 1-15) vanishes behind a no-ref stub. Frontier: 600-char previews, 512k-char limit — sub-1% pressure at 30 iterations, previews maximally lossy.

### Compaction (Phase-4 pillar-9 status TODAY)
Not a single path: 5 independent truncators remain (F4). No protected classes honored (dead `preserveOnCompaction`). No post-compaction self-check of size or answer-derivability anywhere. The one honest projector is projectResults' preview+ref; compactHistory's stub is a pointer with no pointer.

### P4 strategy-switch carryover
Carried: 8 successful obs steps (gates only), toolsUsed, synthetic failed-tool steps, handoff→priorContext (unrendered). LOST to the model: entire conversation evidence, plan state, todo *visibility* (list survives in scratchpad but nothing re-surfaces it; model must call `todo` again), pendingGuidance. See F5.

### Memory package within a run
Only across runs (F7). Bootstrap content is frozen at run start; live semantic writes invisible in-run; RecallService seam exists but is Noop-defaulted and its outputs voided.

### todo meta-tool (P6a)
Best-in-class persistence of anything audited: `_todo:<taskId>` in the shared scratchpad Ref (`act/meta-tool-handlers.ts:144-158`), cleared on fresh non-resume runs (`runner.ts:257-265`), folded into `KernelState.scratchpad` at act end-of-round → checkpoint/crash-resume safe, untouched by compaction, survives strategy switch (process-global Ref + taskId key). Two soft spots: (a) list content reaches the model only as a tool result subject to normal preview aging — no ambient re-render of the current checklist; (b) a checkpoint taken between switch-reset and the next act-round merge snapshots an empty state.scratchpad (narrow race, resume would drop the list if the process also died).

## Concrete failure hunts

**1. Iteration-30 recall of an iteration-3 fact (default config, mid tier).** Mechanisms: (a) visible window — YES but only the first 1,200 chars/heading skeleton of that result (project-results preserve cap); the fact survives iff it's in the skeleton; (b) scratchpad `_tool_result_N` — full fact present, model-unreachable (recall gated off F2; write_result_to_file returns no content); (c) recall tool — hidden (F2); (d) semantic memory — default OFF, and write-only in-run even when on (F7); (e) extractedFact — only if the fact is a dollar amount/percentage/URL, and then only to the grounding corpus, not the model. **Default answer: the fact is available iff it survived the 1,200-char preview cut at iteration 4. Otherwise the run cannot recover it without re-calling the tool.**

**2. Re-read tax.** Nothing prevents or detects non-consecutive duplicate gathering (F6). Loop detector catches only N-in-a-row same tool+args (loop-detector.ts:44-60). No result cache, no dedup nudge, no telemetry counter for re-gathered evidence.

**3. Cross-phase deliverable references (rw-8 shape).** plan-execute/blueprint: phase-1 constraints live in `PlanStep.result` (preview) + `PlanStep.fullResult` (full, plan-execute.ts:708-710) for the whole plan; a phase-5 step reaches them via `{{from_step:s1:full}}` → `fullResult` (fixed, `types/plan.ts:190`) and synthesis reads `fullResult ?? result`. **Reactive/reflexion/ToT have no equivalent**: by phase-5-equivalent iterations, phase-1 constraints exist only as the ≤preserve-budget preview (or post-compaction nothing) — a constraints document >1,200 chars is skeleton-only for the entire rest of the run. The blast radius of report-06's rw-8 concern is thus reactive long runs, which is also the default strategy.

**4. from_step coverage.** Present: plan-execute + blueprint (shared `resolveStepReferences`, `types/plan.ts:172-205`; bare ref 380 chars, `:summary` 500 distilled, `:full` = fullResult post-0.5-2). Absent: reactive, reflexion, tree-of-thought, code-action, direct — no reference syntax at all; their only cross-iteration data channel is the (lossy) window plus the (unreachable) scratchpad. The `result_ref` vocabulary in previews is act-on-by-reference (`write_result_to_file`) only — there is no `read_result`/`from_ref` splice for non-plan strategies.

## Evidence-retention requirements for the RunLedger (Phase 4)

Each failure class above maps to a ledger entry type + projection:

1. **EvidenceEntry (per tool result): `{id/ref, tool, args-hash, full, preview, extractedFact, storedKey, iteration}`** — append-only; every store today (messages content, scratchpad, PlanStep.result/fullResult, semantic write) becomes a projection of this one entry. Kills the dual retrieval vocabulary (F2): ONE ref format, rendered identically in previews, recall, and from_step.
2. **Model-facing retrieval projection**: a `read_result(ref, [range])` capability (or recall keyed by the SAME `result_ref` shown in previews) that is surfaced whenever any preview+ref exists in the window — gate on ledger state ("any entry projected as preview this render"), not on a regex over rendered text (fixes F2's marker mismatch permanently).
3. **CompactionRecord: `{droppedRefs[], keptRefs[], stubText, sizeBefore/After}`** — compaction must re-project *from the ledger* and its stub must enumerate dropped refs + one extractedFact line each (an index, not a lie), with a post-compaction size assertion (pillar 9). Honor `preserveOnCompaction` (or delete the flag).
4. **HandoffEntry (strategy switch): `{handoffText, carriedRefs[], toolsUsed}`** — and a renderer: the assembly must have a `priorContext`/handoff stage so switch handoffs, ToT plans, and memory bootstrap reach the prompt (fixes F1). Carried evidence = refs into the same ledger, so the new strategy's window can re-project the top-K previews instead of restarting blind (F5).
5. **GatherEvent index keyed by (tool, args-hash)** — before executing, consult the ledger; on duplicate, return the stored EvidenceEntry (or at least surface "already gathered at iter N, ref=X") instead of re-executing (fixes F6/re-read tax) — plus a `duplicateGathers` counter for the receipt.
6. **DeliverableConstraint entries** (rw-8): phase-outputs/constraints written once, referenceable by ANY strategy via the same ref syntax — from_step generalizes to from_ref across reactive runs (fixes hunt 3/4 asymmetry).
7. **In-run memory projection**: route live semantic writes through the ledger and make `find(scope:memory)` query ledger + live memory, not the frozen bootstrap snapshot (fixes F7).

The scratchpad already IS the full-content store and post-conditions already call `state.steps[]` "the run LEDGER" (report 06) — Phase 4's job is to unify refs/vocabulary, add the model-facing read path, and make compaction/handoff re-projection ledger-driven rather than destructive.
